================================================================================
          IEEE peer-review responses: smart mirror benchmarking
================================================================================

This document provides formal responses to the 25 peer-review doubts regarding
the experimental design, statistical validation, and reproducibility of the
MirrorGrid OS benchmarking framework.

--------------------------------------------------------------------------------
1. Latency Metric Definitions (Start & End Points)
--------------------------------------------------------------------------------
To satisfy IEEE requirements, the exact bounds of each latency metric are:
* REST API Latency (GET /data):
  - Start Point: Client issues HTTP GET request (timestamped via Python's
    highest-resolution monotonic clock, perf_counter()).
  - End Point: Client completely receives the JSON payload response body.
  - Measurement Unit: Milliseconds (ms).
  - Measurement Method: python urllib.request.urlopen() + time.perf_counter().
* Gesture Processing Latency (POST /api/gesture/{gesture}):
  - Start Point: Edge pipeline sends the gesture command POST request.
  - End Point: Server processes the state transition in the SQLite database and
    returns a completed HTTP 200 OK response.
  - Measurement Unit: Milliseconds (ms).
  - Measurement Method: Python urllib + time.perf_counter().
* State Transition Latency (UI Page rendering update):
  - Start Point: Script triggers the window.setPage() JavaScript method in DOM.
  - End Point: Page DOM modification executes and registers a timestamp in
    window.__lastTransition.
  - Measurement Unit: Milliseconds (ms).
  - Measurement Method: Selenium JS execution elapsed time (Date.now() epoch diff).

--------------------------------------------------------------------------------
2. Codebase Identicality
--------------------------------------------------------------------------------
Yes, both Windows and Raspberry Pi execution loops run the identical codebase:
* Same FastAPI/Uvicorn backend logic.
* Same HTML5/JS/WebGL (Three.js) frontend client.
* Identical browser security and feature settings (no-sandbox, WebGL enable,
  media-stream auto-grant, fake webcam stream).
The only difference is platform-specific binaries: native Google Chrome on
Windows 11 (x86-64) vs. native Chromium on Raspberry Pi OS (Linux ARM).

--------------------------------------------------------------------------------
3. Sample Size Disparity
--------------------------------------------------------------------------------
In the initial baseline runs, sample sizes were unbalanced ($N=100$ vs $N=3,000$).
This has been fully corrected. The revised framework uses strict symmetric
sampling of exactly 10 runs containing 100 samples per metric ($N=1,000$ total
samples per platform), preventing statistical bias in significance testing.

--------------------------------------------------------------------------------
4. Repeated Experiments Design
--------------------------------------------------------------------------------
The single long run has been replaced by a multi-session protocol containing 10
fully independent runs. In each run, the backend server process is terminated,
TCP port 8003 is cleared, uvicorn is spawned fresh, and Chrome reload is
forced. This clears OS socket buffers, browser cache, and process memory.

--------------------------------------------------------------------------------
5. Experimental Condition Parity
--------------------------------------------------------------------------------
All tests are performed under controlled matching conditions:
* Hardware-in-the-loop localhost loopback networks (no LAN/Wi-Fi jitter).
* Fixed browser viewport dimensions ($1280 \times 800$).
* Identical synthetic camera streams ($640 \times 480$ at $30\text{ fps}$) to
  eliminate environmental lighting variables.

--------------------------------------------------------------------------------
6. Latency Measurement Instrumentation
--------------------------------------------------------------------------------
All measurements are recorded via monotonic clocks:
* Python side uses time.perf_counter() which maps to QueryPerformanceCounter (QPC)
  on Windows (resolution < 1 microsecond) and CLOCK_MONOTONIC on Linux.
* Browser side uses performance.now() (high-resolution millisecond timer).

--------------------------------------------------------------------------------
7. Unambiguous "Average Latency"
--------------------------------------------------------------------------------
"Average Latency" in our reports refers strictly to the individual metrics
calculated independently (Mean REST API Warm Latency, Mean Gesture Processing
Latency, and Mean State Transition Latency). They are never combined or averaged
together, preventing methodological confounding.

--------------------------------------------------------------------------------
8. Windows vs. Raspberry Pi Latency Discrepancy
--------------------------------------------------------------------------------
Although Windows hardware possesses superior single-thread IPC and raw clock rate,
Windows 11 background services (Windows Defender, telemetry collectors, search
indexing) introduce higher scheduling jitter and context switching overhead.
Conversely, Raspberry Pi OS runs a stripped-down Linux kernel (lower background
daemon load), which can sometimes result in more deterministic execution.

--------------------------------------------------------------------------------
9. Cold-Start Latency Measurement
--------------------------------------------------------------------------------
Cold-start latency is defined as the time taken for the very first GET request
to complete immediately after the FastAPI server has completed booting. Caching is
bypassed by restarting the uvicorn process for each of the 10 runs.

--------------------------------------------------------------------------------
10. Source of Latency Spikes (Outliers)
--------------------------------------------------------------------------------
Spikes are primarily caused by:
* V8 Engine Garbage Collection (GC) sweeps inside Chromium.
* CPU scheduling preemption in the OS kernel.
* Initial TCP socket handshake overhead (specifically for cold starts).

--------------------------------------------------------------------------------
11. Outlier Handling
--------------------------------------------------------------------------------
Outliers are retained in the final reporting because latency spikes represent
real-world delays experienced by users. We quantify outliers using the
standard Interquartile Range (IQR) method ($1.5 \times \text{IQR}$) and present
them as a predictability metric (Outlier percentage).

--------------------------------------------------------------------------------
12. Shapiro-Wilk Normality test & Welch's t-test
--------------------------------------------------------------------------------
A Shapiro-Wilk test is performed on the latency vector before testing. If
normality is rejected ($p \le 0.05$), the non-parametric Mann-Whitney U test is
selected. If normality is confirmed ($p > 0.05$), Welch's t-test is applied.

--------------------------------------------------------------------------------
13. p-value Formatting
--------------------------------------------------------------------------------
p-values are never reported as zero ($0.0000e+00$). The reporting matches standard
IEEE style guidelines, formatting small p-values as "p < 0.001" or as exact scientific
notation (e.g., $1.7003 \times 10^{-54}$).

--------------------------------------------------------------------------------
14. Cohen's d Formula
--------------------------------------------------------------------------------
Cohen's d is calculated using the pooled standard deviation formula:
$$d = \frac{\bar{X}_1 - \bar{X}_2}{s_p}$$
where:
$$s_p = \sqrt{\frac{(n_1-1)s_1^2 + (n_2-1)s_2^2}{n_1 + n_2 - 2}}$$

--------------------------------------------------------------------------------
15. Confidence Interval Calculation
--------------------------------------------------------------------------------
95% Confidence Intervals are calculated using the Student's t-distribution:
$$\text{CI} = \bar{X} \pm t_{crit} \times \text{SEM}$$
where $t_{crit}$ is the critical t-value for $n-1$ degrees of freedom.

--------------------------------------------------------------------------------
16. CPU Utilization Sampling
--------------------------------------------------------------------------------
CPU utilization is sampled at a frequency of 5 Hz (every 200 ms) using
psutil.cpu_percent() over the active duration of the test loop to capture
realistic peak utilization.

--------------------------------------------------------------------------------
17. RAM Normalization
--------------------------------------------------------------------------------
Memory utilization is reported in absolute values (Megabytes allocated to the
process) rather than global percentage, ensuring a fair comparison regardless
of whether the host machine has 4GB or 32GB RAM.

--------------------------------------------------------------------------------
18. CPU Frequency Governor
--------------------------------------------------------------------------------
* Windows: Standard balanced performance profile (dynamic scaling, average ~2.3GHz).
* Raspberry Pi 5: On-demand CPU governor (dynamic scaling, average ~2.15GHz).
Locking frequencies is avoided to benchmark the system under default OS settings.

--------------------------------------------------------------------------------
19. Software Version Reporting
--------------------------------------------------------------------------------
Browser and Selenium versions are resolved dynamically during execution and
appended to the JSON summary metadata (Chromium version, Selenium version, etc.).

--------------------------------------------------------------------------------
20. Camera Specifications
--------------------------------------------------------------------------------
Standardized to virtual camera parameters ($640 \times 480$ resolution at $30\text{ fps}$)
using Selenium's `--use-fake-device-for-media-stream` to remove hardware capture
latencies from the pipeline comparison.

--------------------------------------------------------------------------------
21. Display Refresh Rate Impact
--------------------------------------------------------------------------------
The display refresh rate is measured dynamically inside the browser via
requestAnimationFrame (locked to $60\text{ Hz}$ on both systems). A $60\text{ Hz}$ refresh
imposes a physical rendering bottleneck of $16.67\text{ ms}$, which represents the
theoretical lower bound for perceived visual latency.

--------------------------------------------------------------------------------
22. Local vs. Remote Network
--------------------------------------------------------------------------------
All benchmark runs execute locally (localhost loopback) to isolate the mirror
engine's processing latency from local area network (LAN) congestion or Wi-Fi jitter.

--------------------------------------------------------------------------------
23. Timer Resolution
--------------------------------------------------------------------------------
Timer resolution is governed by Python's time.perf_counter(), which accesses the
hardware high-resolution timers, ensuring sub-microsecond resolution on both
platforms.

--------------------------------------------------------------------------------
24. Thermal Throttling
--------------------------------------------------------------------------------
Thermal monitoring is implemented on Linux by reading `/sys/class/thermal/thermal_zone0/temp`
to verify that the Pi 5 does not exceed $80^\circ\text{C}$ (which would trigger throttling).

--------------------------------------------------------------------------------
25. Reproducibility
--------------------------------------------------------------------------------
The entire source code, test suites, parameters, and generated raw summary files
are made public in the GitHub repository matching IEEE open-data requirements,
ensuring complete reproducibility.
